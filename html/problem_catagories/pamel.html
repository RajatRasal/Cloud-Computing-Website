MapReduce

Why MapReduce came about

Many implementations of special-purpose computation algorithms have been designed previously to process immense datasets such as (in the case of Google) “crawled documents, web request logs, etc.” in order to derive data such as “inverted indices, various representations of the graph structure of web documents, summaries of the number of pages crawled per host, the set of most frequent queries in a given day, etc.” (2). For these computations to be carried out within a short time, the input data must be distributed across hundreds or thousands of machines in a process called parallel computation (2). This triggered the need for a new cloud computing framework: MapReduce.

The initial conditions for a map/reduce framework to be useful is that (17):

  • The algorithm to be run on the dataset can be composed (such that you can combine results of the algorithm mapped on subsets together – aggregation).
  • The result produced is smaller than the initial set of data.
  • Calculation throughout mapping the algorithm on the data set does not depend on external data (only data provided by user is required).
  • The dataset is large enough that splitting it into subsets to distribute for independent computations will not weaken computation performance.

As a result of these conditions, the MapReduce framework came about. It is used for parallel batch processing of data-intensive problems that require arbitrarily dividing the workload into smaller workloads of equal/close size so that they can be distributed across virtual machines for the algorithm to work on the subsets and then combine the results of those computations (3). This parallelisation of computation allows users to achieve fast performance on large-scale clusters on commodity hardware and allows programmers with zero to minimal experience in distributed/parallel systems to exploit the large availability of resources of the cloud (2).

How the framework is utilised

The motivation behind MapReduce allows it to be applied to many different fields such as “data clustering, machine learning, and computer vision” where the map/reduce algorithm is prevalent so that the framework can deal with the parallelism and the mapping of the reduction algorithm will complete the iterations to finalise a result (7).

Google in particular have shown enthusiasm for the programming model as the model is straightforward to incoming programmers as it hides most of the work done (parallelisation, fault-tolerance, locality optimisation and load balancing) and its simplicity on being applied to many problems that can be expressed in terms of the conditions required (2). MapReduce (due to its large scalability and extensive fault tolerance) is also utilised by Google to handle data mining, machine learning and “large-scale web search applications” to Google Search (such as extracting specific data from a large dataset of indexed web pages) (5) (9) because the model has economically benefitted the company in terms of efficient use of data centre resources (2).

Limitations of MapReduce

MapReduce, as we have discussed, is beneficial for when we went to compute on large datasets in parallel however there are certain instances in which this framework is disadvantageous.

Since the framework is built to deal with acyclic data flow, MapReduce cannot be applied to recursive problems. A fundamental yet simple example can be calculating the next term in the Fibonacci sequence. MapReduce cannot be applied in this case as the computation of the current value is dependent on the computation of the two previous values (i.e. f(k + 2) = f(k + 1) + f(k)) (16). Calculating this requires generating sub-computations and as a result, rather than reducing the problem, the complexity of the computation increases (17).

Another case in which MapReduce is not efficient is when the dataset in question is small enough that dividing the workload is not even required to efficiently compute the result (16). If the dataset is small enough to fit on a single machine, performing a single reduce(map(data)) operation will be more efficient rather than going through the whole MapReduce process requiring for the data to be partitioned (17). The MapReduce framework should be used for concurrency purposes only and that subjects it to parallel processing of large data sets.

Spark

Why Spark came about

Spark is a cluster-computing framework motivated in overcoming the limitations of the MapReduce cloud computing paradigm. Meanwhile retaining the scalability and fault tolerance of its inspired framework, Spark compensates for the linear data flow of MapReduce by incorporating iterative machine learning algorithms and adopting interactive data analysis tools and this allows the new framework to reuse datasets across parallel operations to allow cyclic dataflow (1). The difference between the two is that Spark is able keep things in RAM where data is easily accessible, whereas MapReduce shuffles data in and out of disk memory (8). This in effect results in the programs running 100x faster than MapReduce in memory (or 10x faster on disk) (9).

Spark also introduces some new utilities, most importantly, integrated libraries to allow “machine learning, streaming, graph programming and SQL” (8). This is an added benefit compared to MapReduce as it allows the Spark framework to be more generalised than its restricted counterpart. The libraries increase the possible processing functions to work on the dataset and allows other explorative processing tasks into a working pipeline (as detailed in the next section) to work over the model to help tasks in relation to big data meanwhile increasing usability and performance (2).

Limitations of Spark

Although it is much more of an efficient and generalised cloud computing framework than MapReduce, Spark still has drawbacks that make it ineffective for usage (6):

  • No file management system – Spark does not have its own file manager so usually Spark will need to merge with Hadoop (Google’s open source MapReduce framework) Distributed File System (HDFS) and another cloud-based data platform.
  • No support for real-time processing – Spark Streaming is close to but is not real-time processing as the live data gets divided into smaller workloads that get processed and the results are returned in batches (micro-batch processing).
